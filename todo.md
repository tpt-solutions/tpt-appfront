## Phase 5 — Day 5: Compile-Time Optimizer Macro (`tpt-appfront-macros`)
- [x] `#[appfront::component]` proc macro implemented (`tpt-appfront-macros/src/lib.rs`, re-exported as `tpt_appfront_core::component`) — wraps a `UITree`-returning fn and auto-fills `meta.class` (kebab-cased fn name) and `meta.ai.description` (from the doc comment) on the root node if unset
- [x] Static/dynamic detection **and** the Svelte-style compile-time codegen it enables, now both implemented (was previously just the heuristic flag with nothing consuming it):
  - `#[appfront::component]` still uses the token-level heuristic (`tpt-appfront-macros/src/lib.rs::body_reads_signal` scanning for `Signal`/`.get()`/`create_effect`/route-signal markers) since it has no type info at macro-expansion time.
  - `view!` now computes `is_dynamic` **precisely** instead of heuristically: `node_is_static` (`crates/tpt-appfront-macros/src/view.rs`) recursively checks every attribute/text value is a literal (`Expr::Lit`), not a `{ expr }` interpolation. Any provably-static node (and its whole static subtree) is hoisted at codegen time into a call to the new `tpt_appfront_core::static_tree::static_node(id, || ...)` (`crates/tpt-appfront-core/src/static_tree.rs`): a thread-local `HashMap<u64, Rc<dyn Any>>` cache keyed by the address of a per-callsite `static` sentinel, so the subtree is built exactly once and every later render just clones the cached `UITree`. New `ContainerBuilder::new`/`into_only_child`/`with` helpers (`crates/tpt-appfront-core/src/ui_tree.rs`) support extracting/reinserting the cached one-off subtree. A dynamic root can still have individually-cached static children (`parent_static` threading in `gen_node_stmt`/`gen_children`).
  - The flag is now consumed: `tpt-appfront-dom`'s `hydrate_node` (Phase 9) already skipped listener attachment on inert `is_dynamic == false` subtrees; `view!`'s static hoisting now also skips *rebuilding* those subtrees every render, not just skipping hydration work on them.
  - Tests: `crates/tpt-appfront-core/tests/view_macro.rs` (`fully_static_view_is_flagged_static`, `dynamic_view_is_flagged_dynamic`, `static_subtree_inside_dynamic_root_still_builds`, `static_view_is_reproducible_across_calls`) plus unit tests in `static_tree.rs` (build-runs-once, independent ids, cached-clone independence)
- [x] `view!`/`rsx!` templating macro — `crates/tpt-appfront-macros/src/view.rs` implements `view!` (re-exported as `tpt_appfront_core::view`), an HTML-like template macro covering `Container`/`Heading`/`Text`/`Button`/`Input` with token-span error reporting for unknown tags, missing required attributes (`level`/`on_click`/`value`), and child-shape mistakes. It expands to the same `ContainerBuilder` chain you'd hand-write, so there's no hidden runtime cost or separate AST. Tests in `crates/tpt-appfront-core/tests/view_macro.rs`. v1 does not cover `List`/`DataGrid` or two-way binding, and isn't a full Leptos/Dioxus replacement by design

## Phase 13 — Terminal/TUI Backend (`tpt-appfront-tui`) — proves the "one AST, N renderers" thesis further
Positioning: `tpt-appfront-core` already has no opinion on rendering (`tpt-appfront-html`, `tpt-appfront-canvas`, `tpt-appfront-ai-schema` are all independent interpreters of the same `UITree`). A `ratatui`-based backend would let one codebase ship as a web app, a native window, *and* a terminal app — a claim no DOM-rooted framework (Leptos/Dioxus/Yew) can make without a rewrite of their AST.
- [x] New crate `tpt-appfront-tui`: `NodeKind` → `ratatui` widget mapping (`Container`→vertical layout split, `Text`/`Heading`→paragraph, `Button`→bracketed focusable paragraph, `Input`→editable line, `List`→`ratatui` list, `DataGrid`→`ratatui` table), mirroring how `tpt-appfront-canvas/src/paint.rs` maps `UITree`→`egui`. Rendering is pure and headless-testable via `ratatui`'s `TestBackend` (`render`/`render_to_buffer`/`buffer_to_string`). Added to the workspace `Cargo.toml`, deps: `ratatui` 0.30 + `crossterm` 0.28
- [x] Keyboard-driven event dispatch (Tab/↑←/↓→ move focus, Enter/Space activate the focused button, Esc quits, typing edits the focused `Input`) back through the same `Msg` dispatch pattern used by `tpt-appfront-dom`/`tpt-appfront-canvas` — implemented as a unit-testable `TuiDriver::on_key` (no TTY needed) plus a `run` loop over crossterm events
- [x] `tpt-appfront dev --tui` / `tpt-appfront build --target tui` CLI wiring, parallel to existing `--target canvas`/`dom` (`InitTarget::Tui` scaffolds a `tui` crate via new `tui_cargo_toml`/`tui_main_rs` templates; `dev --tui` and `build --target tui|terminal` shell out to `cargo run`/`cargo build --release`)
- [x] `examples/counter-tui` — same counter `UITree` as the other examples; builds clean (`cargo build` + `cargo clippy -D warnings`) and is now covered by the CI `examples` job's native-examples step. Verified headlessly via `tpt-appfront-tui`'s `TestBackend` snapshot + keyboard-driver unit tests; an actual interactive terminal run was not driven in this session (no TTY in the sandbox) — recommend `cd examples/counter-tui && cargo run`, Tab to focus `+1`, Enter to increment, Esc to quit

- [x] New crate `tpt-appfront-webview`: thin native window via `wry` + `tao`, no bundled renderer (`crates/tpt-appfront-webview/src/lib.rs`); the `run(WebviewOptions, on_command)` entry point opens the window and serves a `trunk`-built `dist/` directory over an `app://` custom protocol
