## Phase 5 — Day 5: Compile-Time Optimizer Macro (`appfront-macros`)
- [x] `#[appfront::component]` proc macro implemented (`appfront-macros/src/lib.rs`, re-exported as `appfront_core::component`) — wraps a `UITree`-returning fn and auto-fills `meta.class` (kebab-cased fn name) and `meta.ai.description` (from the doc comment) on the root node if unset
- [~] Static/dynamic detection — implemented at component granularity, not full codegen: the macro token-scans the function body for reactive-system markers (`Signal`, `.get()`, `create_effect`, `route_signal`/`current_route`) and sets `NodeMeta::is_dynamic` on the root node (`appfront-macros/src/lib.rs::body_reads_signal`, new `NodeMeta::is_dynamic: bool` field in `appfront-core/src/ui_tree.rs`); tested in `appfront-core/tests/component_macro.rs`. This is a token-level heuristic (no type info at macro-expansion time — can't see through helper fns, a coincidental `.get()` unrelated to `Signal` reads as a false positive) and nothing consumes the flag yet (no backend skips work based on it). Full minimal-creation-code generation (Svelte-style compile-time codegen for static subtrees, vs. Solid-style runtime fine-grained reactivity that Leptos/Dioxus also do) remains unimplemented and is the actual differentiator described below — still TODO
- [ ] `view!`/`rsx!` templating macro — today `#[component]` only auto-fills `meta.class`/`meta.ai.description`/`meta.is_dynamic` (`crates/appfront-macros/src/lib.rs`); there's no HTML-like syntax at all, apps must hand-write the `ContainerBuilder` chain. Scope v1 down to `Container`/`Text`/`Button`/`Input` with good-enough compile error spans, not full parity with Leptos/Dioxus on day one

Highest-leverage differentiator identified so far, now implemented. *(uncommitted in working tree as of this writing — no example wiring it up yet, and no dedicated test file)*
- [x] Bidirectional AI command endpoint — `crates/appfront-server/src/router.rs` now has `POST /command`: a `Command { action: String, params: HashMap<String, serde_json::Value> }` (`serde::Deserialize`) is read via axum's `Json` extractor (rejects malformed JSON with 400 for free), validated (empty `action` → 400; not in `SmartRouterBuilder::allowed_actions` allowlist, if configured → 403), then handed to an app-supplied `on_command(Command) -> CommandResponse` closure (501 if none configured) — the closure is where an app calls its own `trigger_event`/`navigate_to` against its actual `Msg`/state, since the router is generic over `Msg` and has no live app state of its own to dispatch into. Route is body-size-limited to 16KiB via `DefaultBodyLimit` (closes the Phase 12 note below about not retrofitting this after the fact). 6 new tests in `router.rs` cover: no-handler 501, empty-action 400, allowlist-rejected 403, successful dispatch + response body, oversized-body 413. MCP-server auto-wiring (translating `AiMeta` into MCP tool definitions) is still not built — this endpoint is the manual-`Command` layer underneath it

## Phase 13 — Terminal/TUI Backend (`appfront-tui`) — proves the "one AST, N renderers" thesis further
Positioning: `appfront-core` already has no opinion on rendering (`appfront-html`, `appfront-canvas`, `appfront-ai-schema` are all independent interpreters of the same `UITree`). A `ratatui`-based backend would let one codebase ship as a web app, a native window, *and* a terminal app — a claim no DOM-rooted framework (Leptos/Dioxus/Yew) can make without a rewrite of their AST.
- [x] New crate `appfront-tui`: `NodeKind` → `ratatui` widget mapping (`Container`→vertical layout split, `Text`/`Heading`→paragraph, `Button`→bracketed focusable paragraph, `Input`→editable line, `List`→`ratatui` list, `DataGrid`→`ratatui` table), mirroring how `appfront-canvas/src/paint.rs` maps `UITree`→`egui`. Rendering is pure and headless-testable via `ratatui`'s `TestBackend` (`render`/`render_to_buffer`/`buffer_to_string`). Added to the workspace `Cargo.toml`, deps: `ratatui` 0.30 + `crossterm` 0.28
- [x] Keyboard-driven event dispatch (Tab/↑←/↓→ move focus, Enter/Space activate the focused button, Esc quits, typing edits the focused `Input`) back through the same `Msg` dispatch pattern used by `appfront-dom`/`appfront-canvas` — implemented as a unit-testable `TuiDriver::on_key` (no TTY needed) plus a `run` loop over crossterm events
- [x] `appfront dev --tui` / `appfront build --target tui` CLI wiring, parallel to existing `--target canvas`/`dom` (`InitTarget::Tui` scaffolds a `tui` crate via new `tui_cargo_toml`/`tui_main_rs` templates; `dev --tui` and `build --target tui|terminal` shell out to `cargo run`/`cargo build --release`)
- [x] `examples/counter-tui` — same counter `UITree` as the other examples; builds clean (`cargo build` + `cargo clippy -D warnings`) and is now covered by the CI `examples` job's native-examples step. Verified headlessly via `appfront-tui`'s `TestBackend` snapshot + keyboard-driver unit tests; an actual interactive terminal run was not driven in this session (no TTY in the sandbox) — recommend `cd examples/counter-tui && cargo run`, Tab to focus `+1`, Enter to increment, Esc to quit

## Phase 12 — Webview Desktop Shell (`appfront-webview`) — primary "beat Tauri/Electron" target
Positioning: matches Tauri on binary size/startup (both wrap the OS's own webview — WebView2/WKWebView/WebKitGTK — instead of Electron's bundled Chromium+Node), but needs no second JS/npm toolchain at all, since `appfront-dom`/`appfront-html` already render the UI in pure Rust. `appfront-canvas` remains the opt-in path for GPU-bound apps (data viz, editors, games), not the default desktop story.
- [ ] New crate `appfront-webview`: thin native window via `wry` + `tao`, no bundled renderer
- [ ] Host the existing `appfront-dom` WASM bundle (built via `trunk`) inside the webview for interactive apps, reusing Phase 3/9 hydration work as-is
- [ ] Native <-> webview IPC: dispatch `Msg` from JS/DOM events back into Rust app state (reuse `serde_json` state serialization already used for hydration, Phase 9)
- [ ] `appfront dev --desktop-webview` / `appfront build --target webview` CLI wiring (`appfront-cli`), parallel to existing `--target canvas`/`dom`
- [ ] `examples/counter-webview` — same counter `UITree` as the other examples, verify: window opens, click increments, no Node/npm install required anywhere in the build
- [ ] Binary size comparison recorded here against a minimal Tauri + Electron equivalent of the same counter app
- [ ] IPC capability allowlisting — explicit allowlist of `Msg` variants a hosted page can dispatch back to native, not an open bridge (avoids Electron-style IPC/remote-module vulnerabilities)
- [~] Request body-size limit landed alongside `POST /command` (Phase 10: `DefaultBodyLimit::max(16 * 1024)` scoped to that route in `crates/appfront-server/src/router.rs`) instead of being retrofitted later. Rate limiting (e.g. `tower-governor` or similar) is still not added — needed before this webview IPC bridge or any additional write route is exposed over the network
- [ ] Stretch: mobile via the same webview shell — `wry`/`tao` have partial iOS/Android support, extend `appfront-webview` there later
