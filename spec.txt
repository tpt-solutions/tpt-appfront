# TPT AppFront: System Design and Architecture Document

## 1. Purpose and Vision

TPT AppFront is a unified, hardware-accelerated, AI-native UI framework designed to eliminate frontend fragmentation and bridge the gap between high-performance applications and SEO-optimized documents.

**The Problem:** Modern frontend development is fractured. Desktop apps rely on heavy WebViews (Electron/Tauri). Web apps rely on heavy JS frameworks and the DOM, which introduces overhead for complex, state-heavy applications. Furthermore, developers are forced to choose between canvas-based rendering (fast, but invisible to search engines) and DOM-based rendering (crawlable, but slow for complex UI). Finally, the current web is built for human browsers, leaving AI agents struggling to crawl, understand, or interact with modern SPAs.

**The Solution:** TPT AppFront allows developers to write their UI and application logic **once in pure Rust** using an abstract UI tree. 
- On **Desktop**, it compiles to a native, lightweight binary using hardware-accelerated rendering.
- On **Web**, it compiles to WASM and renders directly to a WebGPU/WebGL canvas, functioning as a high-performance, offline-capable PWA.
- For **SEO and AI**, it utilizes a Universal Dual-Render Architecture, simultaneously outputting semantic HTML and structured machine-readable data from the exact same codebase.

**Philosophy:** This project is built as a high-velocity, "vibe-coded" proof-of-concept. It is designed to be easily forked, extended, and maintained by the community.

## 2. Target Audience and Use Cases

### What TPT AppFront REPLACES (Target Market)
- Local-first desktop applications (dashboards, admin panels, productivity tools).
- Complex, state-heavy Web PWAs (crypto wallets, data visualizers, SaaS interfaces).
- Internal enterprise tools where performance and small bundle size are critical.
- Marketing sites and documentation that require both SEO (HTML) and interactive elements (Canvas).

### What TPT AppFront DOES NOT REPLACE (Out of Scope for v1)
- Hyper-native consumer mobile apps (iOS/Android app store submissions are deferred).
- Applications requiring deep, pixel-perfect OS-level accessibility integrations (relying on HTML fallback for a11y).

## 3. Core Architecture

TPT AppFront uses a layered architecture centered around an abstract UI tree, abstracting the rendering target while keeping the UI logic identical.

### 3.1 The Abstract UI Layer (The `UITree`)
Instead of directly calling rendering APIs, developers build an abstract syntax tree (AST) of UI components.
```rust
// Example of building the abstract UITree
fn dashboard_ui(state: &AppState) -> UITree {
    UITree::container(|c| {
        c.heading(1, "Dashboard").class("text-2xl font-bold");
        c.data_grid(&state.records).class("w-full mt-4");
        c.button("Export").on_click(Event::ExportData);
    })
}
3.2 The Rendering Backends
The abstract UITree is consumed by specialized renderers based on the target environment:
Canvas Backend: Renders to wgpu via egui for interactive, hardware-accelerated display on Desktop and Web.
HTML Backend: Renders to a semantic HTML string for SSR (Server-Side Rendering), SSG (Static Site Generation), and crawlers.
3.3 The State & IPC Layer
Application state is held in standard Rust structs.
Asynchronous tasks (network requests, file I/O) are handled by tokio (desktop) or wasm-bindgen-futures (web).
Communication between the UI thread and background tasks uses flume or tokio::sync channels, passing strongly-typed Rust enums.
4. Tech Stack
Core Engine
Windowing & Events: winit (Cross-platform window creation and event loop).
Rendering: wgpu (Hardware-accelerated, cross-platform graphics API).
UI Paradigm: egui (Immediate mode UI library, used as the canvas renderer backend).
Typography: cosmic-text (Browser-grade text shaping and layout for the canvas).
HTML Generation: Custom string-builder or markup crate for the HTML backend.
Web & PWA Integration
WASM Bindings: wasm-bindgen and web-sys (Connecting Rust to browser APIs).
Bundler: trunk (Compiles WASM, processes assets, and serves the dev environment).
PWA: Standard Service Worker implementation for offline capabilities.
Application Logic & DX
Routing: routefinder or custom hash-router (for Web URL routing and deep linking).
Serialization: serde and serde_json (State persistence, local storage, IPC, and multi-format output).
Styling: Custom Tailwind-like macro layer wrapping egui styling (e.g., .class("bg-blue-500 p-4")).
CLI Tooling: clap (For the appfront CLI binary).

6. Universal Dual-Render Architecture (The SEO Breakthrough)
TPT AppFront eliminates the dichotomy between canvas-based applications and DOM-based documents by rendering the abstract UITree to multiple formats simultaneously.
How It Works
Smart Routing: A server-side or edge-side router detects the client type via User-Agent or query parameters.
Human Users: Receive the WASM canvas app (fast, interactive, offline-capable).
Traditional Crawlers (Googlebot): Receive pre-rendered, semantic HTML generated by the HTML Backend. This ensures perfect SEO, link-graph crawling, and indexability.
Social Media Bots: Receive pre-rendered OpenGraph meta tags and preview images.
Impact
Developers no longer need to maintain a separate Next.js/Astro site for marketing and a separate React/Canvas app for the dashboard. The entire product—marketing site, docs, and interactive app—lives in one Rust codebase and renders appropriately for the client.
7. AI Era Architecture
TPT AppFront is designed from the ground up to accommodate AI agents, both as consumers of your application and as generators of your code.
7.1 Multi-Format Output for AI Crawlers
Beyond HTML, the UITree can be rendered to machine-readable formats:
JSON-LD: For Google's structured data and rich snippets.
AI Schema: A custom JSON format optimized for AI agent consumption, detailing interactive elements, state, and available actions.
Example: An AI agent can read the AI Schema to understand that a "Add to Cart" button exists, what endpoint it hits, and what parameters it requires, allowing it to interact with the app programmatically.
7.2 AI-First Developer Experience
Predictable Patterns: The UITree abstraction is simple, consistent, and highly pattern-matched, making it incredibly easy for LLMs to generate correct UI code.
Compile-Time Validation: UI code is validated at compile time, reducing runtime errors from AI hallucinations.
State Schema Exposure: AI agents can query the application's Rust data structures to generate UI that correctly utilizes existing state.
7.3 Programmatic API for AI Agents
TPT AppFront exposes a headless, programmatic API that AI agents can use to interact with the application without visual recognition or brittle HTML selectors:
// AI agent can query and manipulate the app state directly

appfront::trigger_event(Event::ExportData);
8. Performance Profile
TPT AppFront is highly optimized for interactive applications, but acknowledges the DOM's supremacy in pure document rendering.
Where it DOMINATES: Complex state-heavy UI, real-time data grids, hardware-accelerated animations, and memory-constrained environments. It is significantly faster and uses a fraction of the memory compared to Electron, Tauri, or heavy React SPAs.
Where DOM is faster: Simple static text rendering and native text selection.
The Solution: The Universal Dual-Render architecture ensures that pure documents are served as HTML (leveraging the browser's 30-year-optimized DOM), while interactive applications are served as Canvas (leveraging the GPU).




Automating the Optimizations (The AI Approach)
This is where TPT AppFront can be genuinely innovative. Instead of manually optimizing, you can automate the optimization process using compile-time analysis and AI.
A. Compile-Time Optimization Pass
Analyze the UITree at compile time and automatically apply optimizations:
// Macro that analyzes the UITree and optimizes it
#[appfront_optimize]
fn my_ui() -> UITree {
    UITree::container(|c| {
        c.text("Static content").class("large");  // Detected as static → texture cache
        c.button("Click me").on_click(handler);  // Detected as interactive → canvas
        c.data_grid(&state.items);               // Detected as dynamic → virtual scroll
    })
}
What the macro does:
Analyzes the UITree structure
Identifies static vs dynamic content
Identifies repeated elements (cache candidates)
Identifies large lists (virtual scroll candidates)
Generates optimized rendering code
B. Runtime Profiling + Auto-Tuning
Profile the app at runtime and automatically switch rendering strategies:
struct AutoOptimizer {
    frame_times: Vec<f32>,
    strategy: RenderStrategy,
}

impl AutoOptimizer {
    fn update(&mut self, frame_time: f32) {
        self.frame_times.push(frame_time);
        
        // If frame time is too high, switch to more aggressive optimization
        if self.average_frame_time() > 16.0 { // Below 60fps
            self.strategy = RenderStrategy::AggressiveOptimization;
            enable_texture_caching();
            enable_virtual_scrolling();
        }
    }
}
C. AI-Powered Optimization Suggestions
Use AI to analyze the UI structure and suggest optimizations:
// AI analyzes the UITree and suggests optimizations
fn ai_optimize_suggestions(ui: &UITree) -> Vec<OptimizationSuggestion> {
    let analysis = analyze_ui_structure(ui);
    
    let mut suggestions = vec![];
    
    if analysis.has_large_static_text_blocks() {
        suggestions.push(OptimizationSuggestion::TextureCache);
    }
    
    if analysis.has_large_lists() {
        suggestions.push(OptimizationSuggestion::VirtualScroll);
    }
    
    if analysis.has_frequent_rerenders() {
        suggestions.push(OptimizationSuggestion::Memoization);
    }
    
    suggestions
}

## 11. Performance Optimization Strategy

TPT AppFront acknowledges that the DOM is highly optimized for document rendering. Rather than competing directly, TPT AppFront uses a smart, multi-layered approach to performance.

### 11.1 The Performance Reality
- **For documents (blogs, marketing):** DOM is 2-4x faster due to 30 years of browser optimization
- **For applications (dashboards, data grids):** Canvas is 10-100x faster due to direct GPU rendering and no DOM diffing

### 11.2 Optimization Layers

#### Layer 1: Smart Rendering Strategy
The framework automatically chooses the best rendering strategy based on content type:
- Static documents → HTML backend (leverages DOM optimizations)
- Interactive applications → Canvas backend (leverages GPU)

#### Layer 2: Canvas Optimizations
For canvas rendering, TPT AppFront applies automatic optimizations:
- **Texture Caching:** Repeated text is cached as GPU textures (5-10x speedup)
- **Virtual Scrolling:** Large lists only render visible items (constant memory usage)
- **Incremental Rendering:** Only re-render changed UI elements

#### Layer 3: Compile-Time Optimization
The `#[appfront_optimize]` macro analyzes the `UITree` at compile time and automatically applies:
- Static content detection → texture caching
- Large list detection → virtual scrolling
- Repeated element detection → memoization

#### Layer 4: Runtime Auto-Tuning
The framework profiles performance at runtime and automatically adjusts:
- If frame time exceeds 16ms, enables more aggressive optimizations
- If memory usage is high, reduces texture cache size
- Continuous monitoring and adjustment

#### Layer 5: AI-Powered Optimization (Future)
AI analyzes the UI structure and suggests optimizations:
- Identifies performance bottlenecks
- Suggests architectural improvements
- Automated benchmarking in CI/CD

### 11.3 Performance Targets
- **Documents (HTML backend):** Match DOM performance (within 10%)
- **Applications (Canvas backend):** 10-100x faster than DOM-based frameworks
- **Memory usage:** <5MB for typical applications
- **Startup time:** <200ms for WASM, <100ms for native




What a "Better DOM" Actually Looks Like
A better DOM isn't a browser engine (that's a decade-long project). It's a new rendering paradigm that keeps the DOM's strengths (ubiquity, accessibility, SEO) while eliminating its weaknesses.
The 7 Principles of a Better DOM
Fine-Grained Reactivity (No Virtual DOM)
Updates go directly to the specific DOM node that changed
No diffing, no reconciliation, no overhead
Inspiration: SolidJS, but compiled and type-safe
Compile-Time Optimization (Not Runtime)
The framework analyzes your UI at compile time
Generates minimal, optimized DOM operations
Inspiration: Svelte, but more aggressive
GPU-Accelerated Layout
Layout calculations run on the GPU via compute shaders
Main thread stays free for interaction
This is the breakthrough no one has nailed yet
Streaming Hydration (Resumability)
Server sends HTML + serialized state
Client "resumes" without re-rendering
Inspiration: Qwik's resumability
Type-Safe End-to-End
UI is defined in Rust (or a typed DSL)
Compile-time validation of props, events, state
No runtime errors from typos
AI-Native by Design
Every component exposes structured metadata
AI agents can query, navigate, and interact programmatically
Built-in JSON-LD, OpenGraph, AI Schema output
Dual-Output (DOM + Canvas)
Same code renders to DOM for documents/SEO
Same code renders to Canvas for apps/performance
Automatic selection based on content type

The Key Innovations
Innovation 1: Fine-Grained Reactivity in Rust
Instead of React's Virtual DOM, TPT AppFront uses signals (like SolidJS) but compiled to zero-cost Rust.
// Reactive state - no Virtual DOM, direct updates
let count = Signal::new(0);

// UI automatically updates when signal changes
fn counter_ui() -> UITree {
    UITree::button(|b| {
        b.text(format!("Count: {}", count.get()));
        b.on_click(|| count.set(count.get() + 1));
    })
}
When count changes, only the text node updates. No diffing, no reconciliation. This is 10-100x faster than React.
Innovation 2: Compile-Time DOM Optimization
A procedural macro analyzes your UITree and generates minimal DOM operations.
// You write this:
#[appfront::component]
fn user_card(user: &User) -> UITree {
    UITree::div(|d| {
        d.class("card");
        d.text(&user.name);
        d.text(&user.email);
    })
}

// The macro generates this (at compile time):
fn user_card_compiled(user: &User) -> DomNode {
    let div = create_element("div");
    div.set_class("card");
    let text1 = create_text_node(&user.name);
    let text2 = create_text_node(&user.email);
    div.append(text1);
    div.append(text2);
    div
}
No runtime overhead. No Virtual DOM. Just minimal DOM operations.

Innovation 4: Streaming Hydration
Server sends HTML + serialized state. Client "resumes" without re-rendering.
// Server-side
async fn render_page() -> HttpResponse {
    let html = appfront::render_to_html(dashboard_ui());
    let state = appfront::serialize_state(&state);
    HttpResponse::ok()
        .body(format!(r#"<div id="app">{}</div>
                         <script>window.__STATE__ = {}</script>"#, html, state))
}

// Client-side (WASM)
fn resume() {
    let state: AppState = serde_json::from_str(&get_state()).unwrap();
    // No re-render! Just attach event listeners to existing DOM
    appfront::hydrate(dashboard_ui(), &state);
}
